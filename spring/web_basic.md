# 인터넷 네트워크

## 인터넷 통신

- `Line`으로 메시지를 주고 받는다

<br>

## IP(Internet Protocol)

- 지정한 `IP 주소`에 데이터 전달
- `패킷`이라는 통신 단위로 데이터 전달

<br>

**한계**

- `비 연결성` : 패킷을 받을 대상이 없거나 서비스 불능 상태여도 패킷 전송
- `비신뢰성` : 중간에 패킷이 사라지면? 패킷이 순서대로 안오면?
- `프로그램 구분` : 같은 IP를 사용하는 서버에서 통신아는 애플리케이션이 둘 이상이면?

<br>

## TCP, UDP

- `TCP` : 전송 제어 프로토콜
  - 연결지향 - 3 way handshake
  - 데이터 전달 보증
  - 순서 보장
  - 신뢰할 수 있는 프로토콜
- `UDP` : 사용자 데이터그램 프로토콜
  - 데이터 전달 및 순서가 보장되지 않음
  - 단순하고 빠름

<br>

## PORT

- 같은 IP 내에서 프로세스를 `구분`해 줌

<br>

## DNS

- 도메인 명을 IP 주소로 변환
- DNS 서버를 통해 도메인을 받아 옴

<br>

# URI와 웹 브라우저 요청 흐름

## URI

- `Uniform` : 리소스가 식별하는 통일된 방식
- `Resource` : 자원, URI로 식별할 수 있는 모든 것
- `Identifier` : 다른 항목과 구분하는데 필요한 정보
  '''
  scheme://[userinfo@]host[:port][/path][?query][#fragment]
  https://www.google.com:443/search?q=hello
  '''

<br>

## 웹 브라우저 요청 흐름

1. 요청
2. DNS 조회
3. 요청 메시지 생성
4. 요청 메시지 전송(패킷 전달)
5. 응답 패킷 전달

# HTTP

## 모든 것이 HTTP

HTTP로 거의 모든 것을 전송할 수 있음

- 특징
  - 클라이언트 서버 구조
  - 무상태 프로토콜, 비연결성
  - HTTP 메시지
  - 단순함, 확장 가능

<br>

## 클라이언트 서버 구조

`Request`, `Respose` 구조

<br>

## Stateful, Stateless

서버가 클라이언트의 상태를 보존 X
장점 : 서버 확장성이 높음(스케일 아웃)
단점 : 클라이언트가 추가 데이터 전송

<br>

## 비 연결성

`서버 자원`을 효율적으로 사용할 수 있음

<br>

## HTTP 메시지

- `start-line` 시작 라인
  - 요청 메시지 : method SP request-target SP HTTP-version CRLF
  - 응답 메시지 : HTTP-version SP status-code SP reason-phrase CRLF
- `header` 헤더
  - field-name ":" OWS field-value OWS
  - HTTP 전송에 필요한 모든 부가정보
- `empty line` 공백 라인(CRLF)
- `message body`
  - 실제 전송할 데이터

<br>

# HTTP 메서드

## HTTP API를 만들어보자

- 가장 중요한 것은 `리소스 식별`이다.
- 리소스인 명사를 HTTP메서드 행위로 활용해라

<br>

## HTTP 메서드 - GET, POST

- `GET` : 리소스 조회
- `POST` : 요청 데이터 처리, 메시지 바디를 통해 들어온 데이터를 처리하는 모든 기능을 수행
  - 처리할 것을 리소스마다 따로 정해야 함(정해진 것이 없음)
  - 보통 새 리소스 생성, 요청 데이터 처리, 다른 메서드로 처리하기 애매할 때 사용

<br>

## HTTP 메서드 - PUT, PATCH, DELETE

- `PUT` : 리소스 대체, 없으면 생성
  - 클라이언트가 리소스 위치를 알고 식별함
  - 변경분에 대해 리소스를 완전히 대체하기 때문에 주의해야 함
- `PATCH` : 리소스 부분 변경
- `DELETE` : 리소스 삭제

<br>

## HTTP 메서드의 속성

- `안전` : 호출해도 리소스를 변경하지 않는다.
- `멱등` : 한 번 호출하든 두 번 호출하든 100번 호출하든 결과는 같다.
- `캐시가능` : 응답 결과 리소스를 캐시해서 사용해도 되는가?

<br>

# HTTP 상태 코드

## 1XX - 요청이 수신되어 처리중

<br>

## 2XX - 성공

- `200 OK` : 성공
- `201 Created` : 요청 성공해서 새로운 리소스가 생성됨
- `202 Accepted` : 요청 접수되었으나 처리가 완료되지 않았음
- `204 No Content` : 서버가 요청을 성공적으로 수행했지만, 응답 페이로드 본문에 보낼 데이터가 없음

<br>

## 3XX - 리다이렉션

- `영구 리다이렉션` : 특정 리소스의 URI가 영구적으로 이동
  - `301` : 요청 메서드가 GET으로 변하고, 본문이 제거될 수 있음
  - `308` : 요청 메서드와 본문 유지
- `일시 리다이렉션` : 일시적인 변경
  - `302` : 요청 메서드가 GET으로 변하고, 본문이 제거될 수 있음
  - `307` : 요청 메서드와 본문 유지
  - `303` : 리다이렉트 시 요청 메서드가 GET으로 변경
- `특수 리다이렉션` : 결과 대신 캐시를 사용

- `PRG` : Post/Redirect/Get
  - POST 주문 후에 결과 화면을 GET 메서드로 Redirect

<br>

## 4XX - 클라이언트 오류

- `400` : 요청 구문, 메시지 등등 오류
- `401` : 인증되지 않음
- `403` : 서버가 요청을 이해했지만 승인 거부
- `404` : 요청 리소스가 서버에 없음

<br>

## 5XX - 서버 오류

- `500` : 서버 내부 문제로 오류 발생
- `503` : 서버가 일시적인 부하 또는 예정된 작업으로 잠시 요청처리 불가

<br>

# HTTP 헤더

RFC723X

- `Content-Type` : 표현 데이터의 형식 설명
- `Content-Encoding` : 표현 데이터 인코딩
- `Content-Language` : 표현 데이터의 자연 언어
- `Content-Length` : 표현 데이터의 길이

<br>

### 캐시 제어 헤더

Cache-Control(캐시 지시어)

- `max-age` : 캐시 유효 시간, 초 단위
- `no-cache` : 데이터는 캐시해도 되지만, 항상 원 서버에 검증하고 사용
- `no-store` : 데이터에 민감한 정보가 있으므로 저장하면 안됨
  `Pragma` : 캐시 제어(하위 호환)
  `Expries` : 캐시 만료일 지정(하위 호환)
